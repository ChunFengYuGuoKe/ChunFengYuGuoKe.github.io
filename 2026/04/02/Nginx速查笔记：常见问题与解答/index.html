1. Nginx 多进程架构：为什么多个 worker 进程能同时监听同一个端口？

核心机制

Nginx 的多个 worker 进程能够同时监听同一个端口，主要依赖两种底层机制：

方式一：fork() 继承文件描述符（传统模式）

  1. Master 进程启动：先创建监听 socket，执行 bind + listen
  2. 创建子进程：通过 fork() 创建多个 worker 子进程
  3. 继承特性：fork() 会让子进程继承父进程的所有文件描述符
  4. 结果：每个 worker 都拥有一个指向同一个监听 socket 的文件描述符
  5. 连接处理：连接到达时，所有 worker 被唤醒，通过 accept() 争抢，只有一个成功

方式二：SO_REUSEPORT 套接字选项（现代模式，Nginx 1.9.1+）

  1. 独立创建：每个 worker 独立创建自己的监听 socket
  2. 关键选项：绑定前设置 SO_REUSEPORT，告诉操作系统允许不同进程绑定同一端口
  3. 内核级分发：连接到达时，操作系统内核直接决定分发给哪个 worker
  4. 无锁高效：每个 worker 有独立的 accept 队列，无需进程间锁竞争

两种模式对比

特性 fork() 继承模式 SO_REUSEPORT 模式
工作方式 共享同一个监听 socket 每个 Worker 有独立的监听 socket
连接分发者 Worker 进程通过 accept_mutex 锁抢 操作系统内核
性能开销 有惊群效应，需要进程间锁 无惊群，无进程间锁
配置方式 默认开启 需在 listen 指令后添加 reuseport 参数

2. 为什么不使用多线程而是多进程？

核心原因：稳定性、安全性、性能可扩展性

维度 Nginx 多进程模型 传统多线程模型
稳定性 极高。一个 Worker 进程崩溃，不影响其他 Worker，Master 会立即拉起新进程 脆弱。一个线程崩溃可能导致整个进程挂掉
安全性 强。进程间天然隔离，内存无法互相访问 弱。线程间可访问彼此内存，一个线程的错误可能污染其他线程数据
锁竞争 几乎无锁。每个 Worker 独立处理请求 锁竞争严重。共享数据结构需要大量锁
可扩展性 强。可平滑调整 Worker 数量，完美适配 NUMA 架构 受限于全局锁和内存争用，线程数增加性能先升后降
资源开销 进程切换开销稍大，但 Nginx 通过固定进程数 + 长连接复用，开销可控 线程切换开销小，但锁等待和内存管理开销往往更大

为什么这种设计适合 Nginx？

  1. 稳定压倒一切：作为流量入口，必须保证即使某个请求处理出错，也不会拖垮整个服务
  2. 无锁高性能：每个 Worker 独立处理连接，几乎不需要锁，高并发下依然保持低延迟、高吞吐
  3. CPU 亲和性与 NUMA 优化：可将每个 Worker 绑定到特定 CPU 核心，避免跨核心内存访问
  4. 平滑升级与热部署：通过新旧进程逐步迁移连接，实现毫秒级服务重启

3. Nginx 请求处理流程：master 和 worker 的职责是什么？

正确流程

关键点

  • Master 进程：不参与网络 I/O，不分发请求。职责是管理 Worker（启动、停止、重启、监控）
  • Worker 进程：真正接收和处理请求，直接通过 accept() 从内核获取连接

常见误区：Master 进程负责监听端口并分发请求 → 这是错误的！

4. 反向代理如何解决跨域问题？

跨域的本质

浏览器的同源策略规定：当前页面的源（协议+域名+端口）与请求的目标源必须一致，否则请求会被拦截。

Nginx 解决方案

原理：

  • 浏览器访问 example.com/api/xxx → 同源请求
  • Nginx 收到后，内部转发给 localhost:8080
  • 拿到响应后返回给浏览器

关键：浏览器全程只与 example.com 交互，看不到后端地址和端口 → 不存在跨域

5. 前端是如何请求到 Nginx 的？

场景假设

  • 前端部署在 http://example.com（Nginx 托管）
  • 后端运行在 http://localhost:8080

请求路径

  1. 前端代码：fetch('/api/user')
  2. 浏览器自动补全为 http://example.com/api/user
  3. 因为 example.com 的 80 端口由 Nginx 监听，请求到达 Nginx
  4. Nginx 根据 location /api/ 配置，转发到 http://localhost:8080/user
  5. 后端响应 → Nginx → 浏览器

为什么这样就不跨域了？

浏览器视角：页面地址 http://example.com，请求地址 http://example.com/api/user → 同源 → 放行

实际情况：Nginx 在背后偷偷把请求转给了真正的后端

这就是反向代理消除跨域的本质：前端只问 Nginx 要数据，Nginx 偷偷去找后端拿。

6. 网友观点辨析：哪些是正确的，哪些是错误的？

网友1

云服务器部署会遇到跨域、Docker 里为什么要套 Nginx、MongoDB 集群 localhost 不行等问题，这些都是反向代理的应用场景。

✅ 正确。真实场景描述准确，反向代理确实是解决这些问题的核心手段。

网友2

跨域一般是 nginx 或后端代码处理，个人建议后端处理，写个中间件改响应头就行。

⚠️ 部分正确。后端处理 CORS 确实可行，但”不推荐 Nginx 处理”就绝对了。两种方式都可以，取决于架构设计。

网友3

跨域后端可以处理，生产环境一般不是前端和 nginx 解决。

⚠️ 个人经验，非通用标准。很多生产环境恰恰是用 Nginx 统一处理跨域的，尤其是微服务架构。

7. 真实场景案例：云部署中的常见坑点与解决方法

坑点1：本地正常，上云跨域

原因：本地开发框架（Vite/Webpack）自动做了代理，上云后没有代理，前后端端口不同触发跨域。

解决：用 Nginx 做反向代理，让前端请求同源地址。

坑点2：Docker 里跑 Node，为什么还要套 Nginx？

原因：Node 不适合处理静态文件、SSL 证书管理复杂、缺少成熟的反向代理/限流/缓存能力。

解决：Nginx 做”大门”，Node 只做业务逻辑。

坑点3：MongoDB 集群用 localhost 不行，必须公网 IP

原因：副本集成员通信时，会把自己配置中的 host 告诉其他成员。localhost 对其他成员来说指向的是自己，不是真正的节点。

解决：用公网 IP 或内网 IP（如 10.0.0.2），或用 127.0.0.1 替代 localhost。

8. Nginx 配置实践：Web服务器、反向代理、负载均衡

基础配置结构

location 匹配规则

写法 说明 优先级
location = / 完全匹配 最高
location ^~ /static/ 前缀匹配，不再匹配正则 次高
location ~ \.php$ 正则匹配（区分大小写） 中
location / 前缀匹配 最低

常用指令速查

指令 作用
proxy_pass 反向代理目标地址
proxy_set_header 设置转发请求头
upstream 定义后端服务器集群
try_files 按顺序尝试文件/目录/fallback
rewrite URL 重写
return 直接返回状态码或内容

9. Nginx 架构与核心概念总结

架构图

核心能力总结

能力 说明
Web 服务器 托管静态文件，支持 SPA 路由
反向代理 隐藏后端，解决跨域，统一入口
负载均衡 分发请求，动态扩缩容，故障自动剔除
网关 限流、封禁、日志、压缩、WebSocket
协议支持 HTTP、HTTPS、HTTP/2、TCP、UDP

关键记忆点

  1. Master 不处理请求，Worker 才是真正的”干活的人”
  2. 反向代理消除跨域的本质是让浏览器”看不到”后端
  3. 多个 Worker 可以同时监听同一端口，依赖 fork() 继承或 SO_REUSEPORT
  4. Nginx 选择多进程而非多线程，是为了极致的稳定性、安全性和可扩展性
  5. 配置文件是核心，理解 http → server → location 的层级结构
