Sinks

概述

Sinks 是 Reactor 中用于手动发射数据的组件，支持通过 tryEmit* 方法主动推送数据。通过 fluxView = sink.asFlux() 可将 Sink 转换为 Flux 供下游订阅使用。大多数情况下，直接使用 Mono.fromRunnable() 就足够了。Sinks.empty() 主要用于需要解耦信号的生产和消费的场景，特别是在桥接非响应式代码、多订阅者等待同一信号、或跨组件协调时才有价值。

一、Sink 分类

根据可发射数据数量，Sink 分为两大类：

1. Sinks.One()\ Sinks.Empty()

  • 仅向订阅者发送 一个数据（或完成 / 错误信号）

  • 常用方法：

    • emitValue(T value) → onNext(value) + onComplete()
    • emitEmpty() → onComplete()
    • emitError(Throwable t) → onError(t)

  • 使用场景：

    • Sinks.one()：适用于单值响应的场景，如 RPC 调用、异步结果返回，例如将将回调式 API 转换为响应式 API

    • Sinks.empty()：适用于只关心操作是否完成，不关心返回值的场景

      • 实际上这里似乎也可以使用用Mono.fromRunnable().doOnError()来实现，为什么不这样做呢？什么时候需要Sink？
        1. 答案是当需要使用跨线程/跨时序的信号传递时
        2. 此外从生命周期来看，它们也不相同，Mono.fromRunnable：生命周期明确，从创建到完成在一个管道中Sink：生命周期可以跨越多个不同的 Mono/Flux

2. Sinks.Many()

  • 可向订阅者发送 多个数据
  • 根据对历史数据的处理方式，又分为以下三类：

二、Sinks.Many 的三种类型

1. multicast() — 多播

  • 行为：仅向订阅者发送新到达的数据

  • 不保留历史数据，订阅者只能收到订阅之后的数据

  • 支持背压控制

    • directAllOrNothing()
    • directBestEffort()

  • 使用场景1：实时日志广播

  • 使用场景2：非响应式代码与响应式代码的桥接

2. unicast() — 单播

  • 行为：类似多播，但只允许一个订阅者

  • 在第一个订阅者注册之前的数据会被缓冲

  • 可自定义背压缓冲队列：

    • onBackpressureBuffer(Queue)，默认无界

  • 应用场景：适用于只能有一个消费者的场景，如任务队列、独占资源

3. replay() — 重放

  • 行为：为新订阅者重放一定量的历史数据，之后再推送新数据

  • 支持多种重放策略：

    • limit(int)：重放最近 N 条数据
    • all()：重放全部历史数据
    • limit(Duration)：重放指定时间窗口内的数据
    • limit(int, Duration)：同时限制条数和时间范围

  • 应用场景：适用于新订阅者需要看到历史状态的场景

三、其他

  • Sinks.unsafe().many()：不保证线程安全，适用于单线程或已确保同步的场景
