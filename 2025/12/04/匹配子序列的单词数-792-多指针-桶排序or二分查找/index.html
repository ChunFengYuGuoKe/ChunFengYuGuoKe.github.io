题目：https://leetcode.cn/problems/number-of-matching-subsequences/description/

给定字符串 s 和字符串数组 words, 返回 words[i] 中是s的子序列的单词个数 。

字符串的 子序列 是从原始字符串中生成的新字符串，可以从中删去一些字符(可以是none)，而不改变其余字符的相对顺序。

  • 例如， “ace” 是 “abcde” 的子序列。

解法一——二分查找

核心思想：空间换时间 + 有序性利用

1. 预构建索引表（空间换时间）

目的：

  • 将字符串 s 中每个字符出现的所有位置记录下来
  • 建立：字符 → 位置列表 的映射

2. 有序位置列表的关键特性

每个字符的位置列表是自然有序的（因为是按遍历顺序添加的）：

  • 'a': [0, 4] 已经是升序
  • 'b': [1, 3] 已经是升序
  • 这个有序性使得二分查找成为可能

3. 贪心匹配 + 二分查找（核心算法）

对于每个单词的每个字符：

  1. 找到该字符在s中大于上一个匹配位置的最小位置
  2. 使用二分查找在有序列表中快速定位

匹配过程：

  1. **匹配 ‘a’**：
    • searchIndex = -1
    • 在 [0] 中找 > -1 的最小位置 → 找到 0
    • searchIndex = 0
  2. **匹配 ‘c’**：
    • 在 [2] 中找 > 0 的最小位置 → 找到 2
    • searchIndex = 2
  3. **匹配 ‘e’**：
    • 在 [4] 中找 > 2 的最小位置 → 找到 4
    • searchIndex = 4
    • 匹配成功！

关键优化点

1. 避免重复扫描s

传统方法对每个单词都要从头扫描s，时间复杂度 O(N×M)。
此方法只需一次扫描s构建索引，然后每个字符用二分查找 O(log K)。

2. 利用有序性

位置列表自然有序，二分查找 O(log N) 比线性扫描 O(N) 快得多。

3. 贪心选择最小位置

每次选择满足条件的最小位置，确保为后续字符留下最大空间。

复杂度分析

  • 预处理：O(S)，S是s的长度
  • 每个单词匹配：O(L × log K)
    • L：单词长度
    • K：字符在s中的平均出现次数
  • 总复杂度：O(S + Σ(L × log K))

与”多指针桶排序”方法的对比

方面 预索引+二分查找 多指针桶排序
思想 空间换时间，二分加速 事件驱动，并行处理
数据结构 HashMap<Character, List> Queue数组
优势 查找速度快，适合字符分布稀疏 处理流程清晰，适合大量单词
劣势 需要存储所有位置，空间较大 需要维护队列状态
适用场景 s中等大小，words较少 s很大，words很多

算法本质

这个算法的本质是：将子序列匹配问题转化为有序序列上的贪心选择问题

  1. 索引化：将线性搜索空间转化为结构化的查找表
  2. 有序化：利用位置的自然有序性
  3. 二分化：用二分查找替代线性扫描
  4. 贪心化：每次选择最小可行解

这是一种典型的预处理+快速查询的算法设计模式，在数据库索引、搜索引擎等领域有广泛应用。

解法二——多指针+桶排序

核心思想

不主动为每个单词在 s 中寻找匹配，而是让 s 的每个字符主动去匹配等待它的单词。

数据结构

java

  • 创建了26个队列（对应26个小写字母）
  • 每个队列存储：等待匹配该字符的单词信息
  • 每个单词信息用 int[]{i, j} 表示：
    • i：单词在 words 数组中的索引
    • j：该单词下一个要匹配的字符索引

算法流程

1. 初始化阶段

java

  • 将所有单词的第一个字符放入对应的队列
  • 例如：单词 “ace” 会放入队列 p['a'-'a'] 中，存储 {单词索引, 0}

2. 遍历 s 进行匹配

java

  • 遍历 s 的每个字符 c
  • 从队列 p[c-'a'] 中取出所有等待字符 c 的单词

3. 处理匹配到的单词

java

  • 如果当前字符是单词的最后一个字符：匹配完成，计数器加1
  • 如果不是最后一个字符：
    1. 将指针移动到下一个字符 (t[1]++)
    2. 将这个单词放入它下一个需要匹配的字符对应的队列中

      • 单词3:”ace” → 完成匹配，res=3

最终结果：res=3（”bb”不是子序列）

复杂度分析

  • 时间复杂度：O(S + ΣL)
    • S：字符串 s 的长度
    • ΣL：所有单词的总长度
    • 每个字符最多被处理一次
  • 空间复杂度：O(N)
    • N：单词数量
    • 最多存储 N 个指针信息

优点

  1. 高效：避免了为每个单词重复遍历 s
  2. 并行处理：s 的每个字符可以同时处理多个等待的单词
  3. 减少冗余：只处理真正需要的匹配

这种方法的本质是反向思考：不是”单词去s中找匹配”，而是”s的字符去通知等待它的单词”。

总结：桶排序思想的体现

  1. 分桶思想：按关键值（下一个字符）将元素分配到不同桶
  2. 桶内聚集：相同条件的元素聚集在同一桶中
  3. 按序处理：按特定顺序（s的字符顺序）处理桶
  4. 重新分桶：处理完后根据新关键值重新分配到其他桶
  5. 稀疏优化：跳过空桶，只处理有数据的桶

虽然这不是标准的排序算法，但它借用了桶排序的核心思想——通过分桶将全局顺序问题分解为局部并行问题，从而：

  • 降低复杂度：从O(N×M)降到O(S+ΣL)
  • 实现并行：多个单词的匹配可以同时进行
  • 按需处理：只在必要时才处理相关数据

这种”桶式分发，事件驱动”的模式在很多高效算法中都有应用，是分布式计算、流处理等领域的重要思想。

核心思想框架：生产者-消费者模式的反向驱动

核心模式

不是让消费者（words）主动寻找资源（s），而是让资源（s）主动通知等待它的消费者。

通用框架的四个步骤

1. 状态抽象

将每个待处理对象抽象为：

  • 标识符：对象的唯一标识
  • 状态指针：当前处理进度
  • 等待条件：下一步需要什么条件才能继续

2. 条件分桶

创建条件等待队列，将对象按下一步所需条件分组存放。

3. 资源驱动

让资源（输入流）主动触发：

  • 每个资源项检查对应条件的等待队列
  • 处理所有等待该资源的对象
  • 更新对象状态，重新放入新的等待队列

4. 完成检测

当对象达到终止条件时，记录完成。

适用场景分类

场景1：多序列匹配问题

特征：一个主序列，多个查询序列，判断是否为子序列

  • DNA序列匹配
  • 日志流中的模式检测
  • 视频流中的动作序列识别

java

场景2：依赖关系解析

特征：任务有依赖关系，需要按特定顺序执行

  • 构建系统的依赖编译
  • 工作流引擎的任务调度
  • 课程学习的先修关系

java

选择这个框架当：

  1. 一个主数据流 + 多个查询/任务
  2. 查询之间有重叠匹配机会
  3. 需要避免重复遍历主数据
  4. 实时性要求高

不要选择这个框架当：

  1. 数据量很小，简单遍历即可
  2. 查询之间完全独立，没有共享匹配
  3. 主数据是静态的，可以预处理索引

模式识别技巧

当你遇到以下关键词时，可以考虑这个框架：

  • “在流中查找多个模式”
  • “多个依赖的并行处理”
  • “事件驱动的状态转移”
  • “避免重复扫描大数据集”
  • “实时匹配和触发”

性能优化变体

  1. 懒加载分桶：只在需要时创建等待队列
  2. 批量处理：一次处理多个等待对象
  3. 优先级队列：按优先级处理等待对象
  4. 超时机制：清理长时间等待的对象

这个框架的核心价值在于将主动查询转变为被动通知，在合适的场景下可以极大提高效率。
